# [Plan] PII 암호화, 감사 로그 및 피드백 보완 구현 계획

사용자 검토 및 보안 감사 요구사항에 따라 개인식별정보(PII)의 강력한 암호화(AES-256-GCM), 백오피스 관리자 권한 및 이력 추적(Audit Log), 네트워크 오류 시 Fallback 대응, 카메라/폴더 멀티 선택 개선 및 기획적 질문에 대한 설계를 포함하는 통합 개발 계획입니다.

## 1. 사용자 검토 피드백 조치 방안 (Q&A)

### ① PII 암호화 규격 준수 여부
> [!IMPORTANT]
> **단순 마스킹뿐만 아니라 데이터베이스 저장 시점의 암호화는 필수입니다.**
*   **조치**: `api/src/utils/crypto.ts`를 신규 구축하여 **AES-256-GCM** 대칭키 암호화/복호화 모듈을 구현했습니다.
*   기사의 `hometax_id` 정보는 DB 저장(`drivers` 테이블) 시점에 즉시 암호화 처리되며, 인증 및 조회 필요 시점에만 복호화되어 동작합니다. 화면에 노출할 때는 BO 서버 단 및 UI 단에서 평문 노출 없이 `kims***`와 같이 강제 마스킹하여 보안을 강화합니다.

### ② TMAP 경로 서버 통신 구체적 방안
*   **방안**: 백엔드에서 실시간 도로 교통망 상황을 연동하여 최적의 목적지 좌표(위도, 경도) 및 추천 사유를 연산한 후, 모바일 브라우저 표준에 대응하는 T-MAP 딥링크 URL(`tmap://route?goalname={목적지}&goallat={위도}&goallon={경도}`)을 클라이언트에 전송합니다. FO 클라이언트는 이를 받아 `window.location.href = tmapIntentUrl`로 트리거하여 스마트폰 내 네이티브 T-MAP 앱을 즉시 호출합니다.

### ③ 외부 API 장애 또는 실시간 데이터 누락 시 FO 노출 방안
*   **대안**: 외부 관제 서버(기상청, 서울시 TOPIS 등) 장애 시 백엔드는 **DB에 캐싱된 최신 핫존 데이터**를 자동으로 리턴합니다.
*   만약 백엔드 서버와의 연결조차 실패할 경우, FO G-PAN 레이더 화면 상단에 경고 배너(**"네트워크 연결이 불안정하여 마지막으로 저장된 오프라인 캐시 기준으로 작동합니다."**)를 표시하고 `localStorage`에 백업된 캐시를 활용해 화면이 깨지지 않고 정상 가동하도록 대응합니다.

### ④ 영수증 폴더 선택 및 카메라 촬영 병행 등록 방안
*   **조치**: 모바일 OS 브라우저에서 `capture="environment"` 속성이 강제되어 있을 시 카메라 촬영만 활성화되는 문제가 발생합니다.
*   `RoadboarderPage.tsx`에서 이 속성을 제거하여, 모바일 기기 클릭 시 시스템이 **"사진 촬영(카메라)"** 또는 **"파일/사진 보관함 선택(폴더)"**을 제안하는 네이티브 시트를 자동으로 출력하도록 수정합니다.

### ⑤ 사주 카드 "최상", "상", "우수" 등급 산정 기준
*   **기준**: 기사의 생년월일(년/월/일)과 주행 당일 날짜(오늘)의 일진(天干/地支) 간의 삼합(三合), 육합(六合), 상생(相生) 및 귀인성(貴人星)의 존재 여부를 해싱 알고리즘을 빌드하여 결정적(Deterministic)으로 연산합니다.
    *   `BEST` (최상): 일진이 기사 사주와 합(合)을 이루고 천을귀인이 들어와 장거리 재물 운이 최고조인 상태.
    *   `GOOD` (상): 상생 관계로 고단가 호출지(예: 강남/청담)와의 매칭률이 높은 상태.
    *   `NORMAL` (우수): 평온한 상태로 출퇴근길 도심 주요 오더 위주로 균형 잡힌 주행이 유리한 상태.
    *   `BAD` (평온/안전): 충(沖)이나 원진(怨嗔)이 끼어 리스크가 있는 날로, 불안감을 최소화하기 위해 '안전 운행 대기' 중심의 안정적인 대기소(예: 공항 슬롯)로 유도하도록 안내합니다.

### ⑥ 영수증 샘플
*   **공유**: 택시 미터기에서 인쇄되어 나오는 일반 영수증 표준 규격(상호, 사업자번호, 승하차시간, 주행거리, 통행료, 지불 금액) 데이터를 OCR 판독 모듈이 식별하도록 사전에 파싱 로직을 모델링하고 있습니다.

### ⑦ 관리자 생성 정보 저장소 및 FO 회원정보 BO 관리 이력
*   **저장소**: 관리자 계정 생성 정보는 메모리나 단순 캐시가 아닌 **실제 PostgreSQL 데이터베이스(`public.admin_accounts`)**에 영구 저장됩니다. (DB가 끊기면 서버 내부의 로컬 JSON DB 파일에 지속 보관됩니다)
*   **회원정보 BO 이력 관리**: FO 가입 기사 정보는 BO 데이터베이스에서 완전 관리 및 동기화됩니다. 추가적으로, **정보 변경 이력 및 변경 권한 부여**를 위해 `public.admin_audit_logs` 테이블을 스키마에 정의하고, 관리자가 정보를 조회/삭제하거나 기사 프로필에 대한 임의 변경을 시도할 때 생성되는 **감사 이력 로그(Audit Logs)**를 기록 및 저장하는 기능을 서버에 탑재합니다.

## 3. 검증 계획 (Verification Plan)

### Automated Tests
*   `npx ts-node api/src/test_workflow.ts` 가동하여 바뀐 DB 암호화 레이어 및 RAG agent 실행 여부 검사.

### Manual Verification
*   **암호화 검증**: 기사 프로필 저장 후 `local_db.json` 또는 pgAdmin/Supabase를 통해 `hometax_id`가 해독 불가능한 암호문(`iv:authTag:ciphertext`)으로 안전하게 인코딩되어 저장되는지 육안 확인.
*   **폴더/카메라 병행 검증**: 모바일 환경에서 영수증 OCR 카메라 촬영 버튼을 눌렀을 때, 카메라만 강제 구동되지 않고 폴더 갤러리나 파일 탐색기 선택 메뉴가 병렬 노출되는지 브라우저에서 검증.
*   **감사 로그 추적 검증**: BO에서 관리자 생성 및 삭제 실행 후, 새로 신설될 감사 로그 뷰에서 변경 이력(어드민 이메일, 변경 일시, 작업 분류, 대상 식별값)이 정확히 저장 및 출력되는지 검증.
